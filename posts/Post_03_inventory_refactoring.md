## 1. 초기 인벤토리의 한계

게임을 처음 만들 때는 단순히 "사과 아이템을 먹으면 사과가 가방에 들어간다" 정도로 생각하기 쉽습니다. 그래서 처음에는 사과라는 아이템 자체를 가방 배열에 그대로 집어넣었습니다.

하지만 게임이 복잡해지면서 문제가 발생했습니다. 같은 '사과'라도 플레이어가 3개를 가지고 있을 때도 있고, 5개를 가지고 있을 때도 있습니다. 또 어떤 사과는 강화가 되어 있거나 내구도가 다를 수도 있습니다. 만약 원본 사과 데이터 자체에 개수나 내구도를 기록한다면, 존재하는 모든 사과의 개수가 동시에 바뀌어버리는 버그가 발생합니다.

## 2. 원본(Master)과 복사본(Instance)의 분리

이 문제를 해결하기 위해, 1차 리팩터링 기간에 **데이터 분리 작업**을 대대적으로 진행했습니다.

- **원본 데이터 (ItemMasterData)**: 도감에 적혀있는 아이템의 기본 정보입니다. "ID 1번, 이름: 빨간 포션, 설명: 체력을 회복합니다." 처럼 절대 변하지 않는 고정된 내용들입니다. 이 데이터는 게임 시작 시 엑셀(TSV) 파일에서 한 번만 읽어와 `InventoryDatabase`라는 중앙 도서관에 보관합니다.
- **실제 데이터 (ItemInstance)**: 플레이어의 가방 칸에 실제로 들어있는 데이터입니다. 원본 데이터가 무엇인지 기억하는 이름표와 함께 "현재 3개 가지고 있음", "레벨 2" 같은 변화하는 상태 값만 따로 저장합니다.

```csharp
// Scripts/Refactoring/Inventory/Inventory.cs 내부 아이템 획득 로직

private void OnTriggerStay2D(Collider2D collision)
{
    // 1. 땅에 떨어진 아이템의 ID를 확인합니다. (예: 1번 아이템)
    int dropItemID = fieldItem.GetItemID();

    // 2. 중앙 도서관(InventoryDatabase)에서 1번 아이템의 '원본 정보(Master)'를 찾아옵니다.
    ItemMasterData master = InventoryDatabase.Instance.GetMasterData(dropItemID);

    if (master != null)
    {
        // 3. 원본 정보에 연결된 '새로운 실물 복사본(Instance)'을 만듭니다. (개수: 1개)
        ItemInstance newItem = new ItemInstance(master, 1);

        // 4. 이 복사본을 내 가방(InventoryList)에 넣습니다.
        if (Items.Add(newItem))
        {
            fieldItem.DestroyItem(); // 땅에 있던 아이템은 없앱니다.
        }
    }
}
```

## 3. 리팩토링의 효과

데이터 구조를 이렇게 분리(`Master`와 `Instance`)한 덕분에 많은 이점이 생겼습니다.

1. **데이터 오염 방지**: 플레이어가 포션을 마셔서 수량을 줄이더라도, 포션의 기본 설명이나 원래 체력 회복량 같은 원본 데이터는 절대 훼손되지 않습니다.
2. **쉬운 데이터 수정**: 아이템의 이름이나 설명을 바꾸고 싶으면 코드나 유니티를 켤 필요 없이, 엑셀 파일(TSV)의 텍스트만 쓱 수정하면 게임에 자동으로 반영됩니다. (어드레서블 텍스트 에셋 활용)
3. **확장성**: 아이템 인벤토리뿐만 아니라, 앞서 설명했던 아군 몬스터(Follower) 인벤토리 시스템도 이 구조를 그대로 복사해서 똑같이 적용(`CharacterMasterData`, `CharacterInstance`)할 수 있었습니다.
